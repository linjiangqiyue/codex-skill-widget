# 产品愿景

## 起点

这是作者的第一个开源项目。它不是从“还能给 AI 加什么功能”出发，而是来自长期使用 Codex 后反复出现的真实困难：

> 我有行业经验和产品想法，但不具备开发能力。我知道结果不对，却不知道它具体错在哪里，也不知道该怎样要求它改对。

## 服务谁

首先服务有想法但缺少开发、产品或 UI 专业表达的中文用户。他们不应该为了使用 AI 编程工具，先学会几百个英文 Skill 名称和复杂的代理编排。

## 核心承诺

1. 把模糊感受翻译成具体问题。
2. 把口语需求整理成有边界、能验收的任务。
3. 不用奉承代替判断，明确区分事实、推断和建议。
4. 不用虚假的进度和时间制造确定感。
5. 复杂逻辑留在后台，前台保持轻量、中文优先。
6. 自动化涉及下载与安装时，安全边界必须清楚可见。

## 产品原则

- **中文优先**：技术名称可以保留，但必须解释它能帮用户做什么。
- **少即是多**：默认只推荐最小能力组合，需要时再浏览完整目录。
- **真实证据**：UI 检查依赖真实截图；完成状态依赖测试和验证。
- **用户掌控**：删除、覆盖、偏离原型等高风险决定必须暂停确认。
- **适配现实设备**：不同屏幕尺寸和缩放比例下都应可用。
- **开放但谨慎**：欢迎社区共同改进，同时不忽视第三方代码风险。

## 暂不承诺

- 无可靠数据时，不展示精确任务进度和剩余时间；
- 不声称可以自动读取或控制所有 Codex 线程；
- 不为了日报、周报、看板等候选功能破坏小组件的轻量体验；
- 不把单一用户的痛点冒充成已经验证的普遍市场结论。

## 希望社区帮助验证的问题

- 这些痛点是否也出现在其他非开发者或中文用户身上？
- 哪些中文解释真正降低了理解成本？
- 三档尺寸能否覆盖常见 Windows 屏幕和缩放设置？
- 自动推荐与完整能力库之间怎样保持最合适的平衡？
- GitHub Skill 搜索和审核还需要哪些安全护栏？
