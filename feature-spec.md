# Codex 普通用户辅助小组件：产品需求文档

## 1. 用户与背景

本产品首先服务一位真实用户：有行业经验、产品想法和审美判断，但没有系统的开发与设计训练。用户会高频使用 Codex 修改 GitHub 源码、实现产品想法和还原原型。

用户原话：

- “我知道它做错了，但是我又不知道它哪错了。”
- “UI 肉眼看像素都不对，但是我没有专业的表达。”
- “Codex 给出来的意见，用户有时候看不懂，就直接点确认了。”
- “我想托管，但是也不知道怎么去写。”
- “后端可以复杂，逻辑可以严谨，最重要的是让用户觉得好用。”

## 2. 问题陈述

当前困难不是缺少更多 AI 功能，而是普通用户无法判断、表达和约束 Codex 的工作：

1. 按 Figma、截图或原型开发时，Codex 会擅自加入自己的设计想法。
2. 产品方案明显有问题时，用户能感觉不对，却无法指出具体原因。
3. UI 存在错位、裁切、间距和字体问题时，用户缺少专业表达，难以形成可执行修复要求。
4. 用户不知道怎样编写可靠的托管提示词，也不知道哪些情况必须要求 Codex 暂停确认。
5. 用户看不懂技术说明时容易直接确认；Codex 又倾向于奉承，缺少独立、具体的风险判断。
6. 用户不知道剩余额度、重置时间，也缺少对任务耗时的基本概念。

## 3. 产品定位

> 把用户“感觉不对但说不清楚”的地方，转换成专业、具体、可以直接交给 Codex 执行的判断与指令。

产品继续采用 Windows 悬浮小组件。前台保持轻量，复杂的 skill 选择、GitHub 搜索、安全审核和提示词组织放在后台。

## 4. v1 目标

- 用户只写自然语言，即可得到结构清楚、包含边界与验收要求的 Codex 提示词。
- 用户能一键切换产品检查、UI 检查、托管任务和 Skill 推荐四种工作模式。
- 每种模式自动选择最小且有顺序的 skill 组合，并解释其用途。
- 小组件持续显示真实额度与重置时间；不可获得时明确说明，不伪造数据。
- 保持单窗口小组件形态，提供小、标准、大三档尺寸，不新增页面，不变成项目管理平台。

## 5. v1 不做

- 不直接声称已经控制或读取其他 Codex 线程。
- 不展示无法验证的百分比进度和精确剩余时间。
- 不自动替用户批准删除、覆盖、新增页面或偏离原型等高风险决策。
- 不实现日报、周报、完整项目看板或复杂代理编排界面。
- 不把更多按钮持续堆叠到首页。

## 6. 核心工作模式

### 产品检查

输出必须区分事实、推断和建议；指出目标、流程、信息结构、范围和风险中的具体问题。禁止只给“很好”“可以优化”等奉承式反馈。

### UI 检查

要求先获取截图或原型，检查裁切、溢出、对齐、间距、字体、圆角、颜色和不同窗口尺寸。输出可直接执行的修改位置、原因与验收方式。

### 托管任务

把口语需求整理为：目标、不可改变项、执行步骤、风险边界、进度汇报节点和验收标准。普通检查自动完成；新增页面、删除覆盖、原型偏离等情况必须暂停。

### Skill 推荐

保留原有中文意图匹配、高频词增强和 GitHub 隔离审核能力，作为高级入口，而不是让用户承担 skill 编排责任。

## 7. 使用流程

1. 用户选择模式，或保留默认的“托管任务”。
2. 用户用口语描述问题。
3. 小组件推荐对应 skills，并显示简短选择理由。
4. 用户点击“复制提示词”。
5. 小组件生成包含约束、验证和通俗汇报要求的提示词。
6. 用户粘贴给 Codex；涉及高风险决策时由 Codex 明确暂停询问。

## 8. 成功标准

- 用户无需知道具体 skill 名称也能生成可用任务。
- 生成的产品检查提示词明确要求反奉承、证据和风险。
- 生成的 UI 检查提示词明确要求截图、像素问题和前后验证。
- 生成的托管提示词包含不可擅自扩展范围及高风险暂停条件。
- 所有模式在 360 × 420、420 × 520、500 × 620 三档尺寸中无横向裁切，并可通过纵向滚动访问完整内容。
- 额度不可用时不展示虚假百分比或虚假倒计时。

## 9. 后续候选

- 当前 Codex 任务的真实阶段和耗时（需要可靠桥接数据后再做）。
- 自动日报与周报。
- 多线程状态汇总。
- 原型与实现截图的自动视觉差异报告。

这些功能只有在不增加首页认知负担、且数据来源可靠时才进入后续版本。

## 10. 当前假设与证据强度

- 需求来自单一高频用户，严重程度和使用情境明确，适合先做个人工具。
- “多数 Codex 用户是否有相同问题”尚未验证，不作为 v1 成功前提。
- 小组件无法直接读取其他线程的可靠进度，因此 v1 不承诺自动托管执行，只生成更可靠的托管任务。
