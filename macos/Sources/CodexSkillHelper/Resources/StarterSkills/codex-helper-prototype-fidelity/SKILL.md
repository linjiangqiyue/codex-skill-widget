# 严格还原原型

把原型视为验收标准，而不是灵感参考。先取得视觉证据，再实现，再用同尺寸截图验证；未通过时继续修正，不得用“代码已完成”代替视觉验收。

## 1. 确认设计源

确定唯一目标及目标状态：Figma frame、设计稿、截图或当前产品界面。记录平台、视口尺寸、缩放比例和交互状态。

- Figma 原型：使用可用的 Figma 设计转代码能力读取节点和设计数据。
- 截图或图片：使用可用的 image-to-code 能力。
- 现有项目：先检查相似页面、组件、设计令牌、字体和图标库。
- 只有文字需求且没有设计源：停止视觉实现，向用户索取原型；不得自行发明视觉方案。

## 2. 建立还原清单

实现前测量并记录：

- 页面和容器宽高、网格、边距、内边距、元素间距；
- 字体、字号、字重、行高、字间距、换行和省略规则；
- 按钮、输入框、图标、卡片的尺寸、圆角、边框和状态；
- 颜色、透明度、阴影、层级、滚动和裁切行为；
- hover、focus、active、disabled、loading、empty、error 等可见状态。

复用项目已有设计系统。不要凭感觉替换字体、图标、颜色、圆角或组件。

## 3. 实现并保护布局

实现主要交互，同时为真实内容建立约束：最小和最大尺寸、换行或省略、滚动区域、安全间距以及不同缩放比例下的行为。禁止用缩小字体掩盖溢出，也禁止通过删除内容让截图看似通过。

## 4. 强制视觉验收

在与原型相同的视口、缩放比例和状态下截图。把原型和最新实现截图放在同一张对比图中检查，不得只分别看两张图。

逐项检查：

- 是否有文字或控件被吃掉、越界、重叠、裁切；
- 按钮、输入框、图标和点击区域是否过大或过小；
- 字体、行高、对齐、间距、颜色、圆角、边框是否一致；
- 窄宽、长文案、系统缩放和核心交互状态是否稳定；
- 键盘焦点、对比度和可操作性是否存在明显风险。

按严重程度记录问题：P0 阻断使用；P1 核心结构或操作错误；P2 明显视觉偏差；P3 轻微润色。

## 5. 修正循环与交付门槛

修复全部 P0、P1、P2 后重新截图并比较，直到通过。项目根目录保存 `design-qa.md`，最后一行必须是以下之一：

- `final result: passed`
- `final result: blocked`

无法取得原型、无法截取实现、视口或状态不一致时标记 blocked，并说明缺少什么。只有 passed 才能声称严格还原完成。交付时用中文报告设计源、实现内容、截图证据、测试结果和剩余 P3。
